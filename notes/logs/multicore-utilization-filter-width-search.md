# Multicore Utilization: Filter-Width Search Bottleneck

**Agent-oriented working log.** Raw, dated investigation notes — not proofread for external readability. For current, human-readable status, see `notes/findings/`.

Date: 2026-07-07

## Context

Following the cyvcf2 VCF I/O migration and the local-search OOM fix (grouped-partition process-pool dispatch, see `docs/optimizations.md` #5), asked whether there were further multicore-utilization improvements available in `ldetect run`.

## Timeline plot revealed an uninstrumented, single-threaded gap

`examples/ldetect_original/plots/EUR-chr21-timeline.pdf` (generated by `scripts/profile_run.py` + `scripts/plot_profile_timeline.py`) showed a large, unlabeled flat span between step 3 (matrix-to-vector) and step 4's "fourier metric" phase: RSS flat at ~260 MiB, CPU flat at ~100% (single core), lasting ~34 of ~64 total seconds for a chr21 EUR run — over half the total wall-clock.

`plot_profile_timeline.py`'s `_PHASES` list only rendered spans with matching `log_memory_checkpoint` start/end pairs, and nothing in `find_breakpoints` (`pipeline.py`) checkpointed the code between `step3_end` and `fourier_metric_start`: reading the vector, `custom_binary_search_with_trackback` (the Hanning filter-width search), and the final `apply_filter`/`get_minima_loc` call. Added three checkpoint pairs (`vector_read_start/end`, `filter_width_search_start/end`, `minima_extraction_start/end`) and matching phases in the plotting script.

Re-running the profiling job confirmed: **`filter_width_search` alone accounts for ~34s**, entirely single-threaded. The re-run log (`plots/EUR-chr21.log`) shows 41 separate `apply_filter` calls in that span (4 exponential search, ~13 binary search, ~24 coarse+fine trackback), each ~1s at widths in the 8,000-18,000 range.

## Root cause: `scipy.ndimage.convolve1d` is a direct (non-FFT) convolution

Cost is O(N·width). Since the exponential search doubles the candidate width and the trackback sweep refines near the converged width (which was ~9,000 for this chromosome), most of the 41 calls fall in the expensive range.

## FFT convolution: tried, reverted — unsafe

Replacing `scipy.ndimage.convolve1d` with `scipy.signal.fftconvolve` (`np.pad(..., mode="symmetric")` to match `convolve1d`'s default `reflect` boundary, verified to agree with direct convolution to ~1e-14 absolute error on random noisy data, and ~100-150x faster at realistic widths) looked like a clean win.

**But it broke 6 of `tests/test_filters.py`'s existing tests**, going from 1 detected minimum to 23 on a synthetic flat-plateau-plus-valley fixture. Root cause: `scipy.ndimage.convolve1d`'s direct convolution produces bit-identical output at every position across a constant/flat input stretch (same shift-invariant arithmetic repeated) — so `argrelextrema`'s strict `<` comparison never fires there. FFT convolution is a *global* transform; its rounding error is not shift-invariant, so it injects distinct ~1e-15 noise at every position, breaking exact ties and manufacturing spurious local minima out of nothing.

Confirmed directly:
```python
arr = np.ones(200) * 5.0; arr[40:61] = <valley>
direct = ndimage.convolve1d(arr, window)  # exactly 5.0 at every flat-region position
# FFT version: same region prints "5. 5. 5. ..." but differs from 5.0 by ~1e-15 per position
```

Since the binary/trackback search is actively hunting for a width producing exactly N minima, spurious minima anywhere would derail convergence to a different `found_width` and a different breakpoint set entirely — real covariance-sum vectors can plausibly have flat/tied runs (sparse regions, zero-padding), so this isn't just a synthetic-fixture artifact. **Reverted in full** (`filters.py` and `tests/test_filters.py` back to the pre-FFT state, confirmed via `git diff --stat` showing no changes).

Other FFT-family options (`scipy.signal.oaconvolve`) share the same global-transform rounding-noise characteristic and would have the same flaw. A numba `prange`-parallel *direct* convolution (same summation structure, hence flat-region-safe by construction) was considered as a bigger-lift alternative but not pursued this session — it would need exact boundary-condition replication and real-data validation to trust as bit-identical to the current output on non-flat data.

## Shipped: thread-parallelize the search's independent-candidate rounds

Confirmed empirically that `scipy.ndimage.convolve1d`/`argrelextrema` release the GIL enough for real `ThreadPoolExecutor` speedup (~5x with 8 threads on 8 independent calls, results identical to sequential).

Two of the three search phases evaluate a boundable/predictable set of candidates per round and can be batched without changing the decision logic at all — same numerics, same final result, just computed concurrently instead of one-at-a-time:

- **`_find_end`** (exponential search, doubling): batches up to `max_workers` doubling candidates per round, returns the smallest satisfying `f(x) < val` — identical to the sequential recursive doubling.
- **`_trackback`** (coarse + fine refinement sweep): batches each round's candidate window in chunks of `max_workers`, applies the original first-match-wins decision rule in the same left-to-right order over the concurrently-computed results.

**Not parallelized**: the core binary search (`find_le_ind`, `_util/binary_search.py`). Each of its steps is adaptive on the previous comparison (can't be pre-batched without a fundamentally different k-ary-search algorithm), and it's a shared utility used elsewhere — changing its core algorithm was judged out of scope for this pass given the added verification burden versus the smaller share of total calls it represents (~13 of 41 in the observed trace).

Wired via a new `search_workers` parameter on `custom_binary_search_with_trackback`, reusing `find_breakpoints`'s existing `workers` argument (already meant for local search, which hasn't started yet at the point the filter-width search runs — no oversubscription risk, confirmed via the flat single-core CPU/RSS in the profiling plot showing no other multiprocessing active during this phase).

Verified via `tests/test_find_minima.py`: `test_trackback_threaded_matches_sequential`, `test_find_end_threaded_matches_sequential`, `test_find_end_threaded_raises_at_max_srch_val` — threaded (`search_workers=4`) vs. sequential (`search_workers=1`) produce identical `found_width` and identical error behavior on a real noisy synthetic vector. Full suite (227 tests), ruff, mypy all clean.

- The binary-search phase itself (~13/41 calls) remains single-threaded.

## Follow-up: real-cluster re-profiling confirmed the threading fix, pinpointed the remaining cost

Asked the user to pull the thread-parallelization commit and re-profile chr21 on the cluster. First attempt still showed no improvement — root-caused (by the user) to a copy-pasted `--output-dir` typo reusing a stale chr2 output directory, not a code issue; the README's own documented command was already correct. A clean re-run's log/plot showed the fix *did* work, exactly as scoped: `filter_width_search` dropped from ~34s to ~20s. Log timestamps showed the exponential+binary-search portion (17 calls, unparallelized by design) taking a fixed ~13s (~1 call/sec), while the trackback portion (24 calls) showed multiple calls landing in the same one-second timestamp repeatedly — direct evidence of real concurrency — completing in ~7s (~3.4x speedup, consistent with `search_workers=4`). This confirmed the ~13s exponential+binary-search portion, un-touchable by thread-parallelization (adaptive, one-call-at-a-time), was now the dominant remaining cost, and motivated the numba pass below.

## Numba direct convolution: implementation and a real snag caught by tests

Planned via Explore + Plan subagents (numba conventions in `shrinkage.py`, scipy `mode='reflect'` boundary semantics, blast radius, GIL-release constraint). Key pre-implementation findings from the Plan agent's own scratch verification: `numpy.pad` is not numba-jittable (must hand-roll the reflect-boundary index formula); the reflect formula was verified bit-exact against `np.pad(mode="symmetric")` including `width >= len(array)` cycling; a GIL-release check via a background counter-thread (not wall-clock speedup, which is CI-flaky) was validated as a reliable technique (~2.7M increments with `nogil=True` vs. ~127K without, during a fixed workload).

Implemented `_reflect_index`, `_pad_reflect`, `_convolve1d_reflect` in `filters.py`, `@njit(nogil=True, cache=True)`, scope limited to this (no `prange`/`parallel=True` — see `notes/findings/multicore-utilization.md` for why that was deferred).

**Caught by the test suite, not assumed away**: `test_apply_filter_larger_width_fewer_minima` failed (20 vs 28 minima, backwards from expected) even though the smoothed array matched scipy to ~2.7e-15 (no logic bug) and the flat-plateau fixtures (`_ARR`/`_ARR2`) still passed exactly. Root-caused: the test's width=2-vs-50 comparison was already an exact tie under *plain scipy* (20==20) — sweeping widths 2-95 under scipy alone showed minima count is genuinely non-monotonic at intermediate widths for this periodic synthetic signal (20, 18, 15, 21, 25, 24, 20, 18, 9, 3). This is a distinct, more general instability than the FFT bug: near-tied *output* extrema (from smoothing structure, not from constant input) are sensitive to any implementation-level floating-point difference, numba or not. De-risked for realistic data: 0 mismatches across 80 randomized noisy-random-walk trials at production-representative widths (up to 9,000). Fixed the test by widening to a decisive width pair (2 vs. 95: 20 vs. 3 under both implementations) rather than weakening the assertion.

**Speed snag, also caught before declaring victory**: initial `nogil=True, cache=True` (no `fastmath`) measured ~2x *slower* than `scipy.ndimage.convolve1d` at realistic widths — LLVM wasn't auto-vectorizing the strict-IEEE-order reduction loop. Adding `fastmath=True` flipped this to ~2x *faster*, and was re-verified not to reopen flat-region risk (fastmath reassociation is fixed at compile time and applied identically at every position — still shift-invariant, still produces bit-identical output on a constant input run, confirmed empirically).

**Final measured result**: ~2x per-call speedup over scipy at realistic widths (8,000-18,000), multiplicative with the already-shipped ~5x threading — combined ~10x over the original sequential-scipy baseline on an 8-candidate batch. Added padding-equivalence, convolution-value-equivalence, exact minima-index-equivalence (scipy vs. numba, not just closeness), degenerate-width, and GIL-release regression tests to `tests/test_filters.py` (12 → 33 tests in that file). Full suite 248 tests, ruff, mypy all clean.

## Follow-up: fixed a bad numba-unavailable fallback before it could bite

User caught that `_njit_nogil`'s `ImportError` fallback (mirroring `shrinkage.py`'s existing pattern) would silently leave `_convolve1d_reflect` as an *un-jitted plain-Python* O(N·width) triple-nested loop if numba were ever missing — ~10^8 iterations at production widths, catastrophically slow, not just non-optimal. Numba is a hard dependency (`pyproject.toml`), so this should be unreachable in practice, but "should be unreachable" isn't the same as "safe if it happens." Fixed: `apply_filter` now checks `_HAVE_NUMBA` and calls the original `scipy.ndimage.convolve1d` directly in that branch, so a broken/tampered environment degrades to "non-optimal" (same as before this whole optimization pass) rather than "unusable." Added `test_apply_filter_falls_back_to_scipy_when_numba_unavailable` (monkeypatches `_HAVE_NUMBA`, since uninstalling numba isn't practical in CI). 249 tests, ruff, mypy clean.

Also investigated, prompted by a direct question, whether `fastmath=True` introduces meaningfully more inexactness than the numba migration already does: measured no-fastmath vs. fastmath maxdiff against scipy directly (both ~1e-13 to ~1e-12 absolute on a signal of scale ~100, fastmath if anything slightly *smaller* here due to FMA's precision gain partly offsetting reassociation) — `fastmath` doesn't add a new risk category, just marginal noise at the same order of magnitude the numba-vs-scipy switch already introduces.

Also investigated whether the `prange`-oversubscription concern (cited above as the reason to defer within-call multicore) is empirically real: composes fine with `nogil`/`fastmath` (no crashes), and a quick local test (unconstrained + a numba-thread-count-limited simulation) showed *no* measurable oversubscription penalty when combined with the outer `ThreadPoolExecutor` — if anything slightly faster. Could not fully replicate a genuinely CPU-affinity-constrained cluster allocation locally (no cgroups/taskset control available), so this isn't a full exoneration, but the real blocker for deferring `prange` is the API-surface refactor cost (per-phase-different filter functions), not a demonstrated performance hazard — worth revisiting if more speedup is wanted later.
