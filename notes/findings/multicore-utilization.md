# Multicore utilization — findings

**Findings summary (current as of 2026-07-08).** Distilled for human review. Full investigation detail, diagnostic scripts, and dated process notes: `notes/logs/multicore-utilization-filter-width-search.md`.

## Status: four fixes shipped, two approaches tried and rejected/superseded, real-cluster validation done, three avenues still open

`ldetect run`'s stage-level parallelism (covariance calc, local search, metric workers) was already well covered. Three real gaps were found and fixed this pass; an FFT-based convolution attempt was found to be numerically unsafe and reverted.

## Shipped

1. **Local-search OOM fix** (`pipeline.py::_run_local_search`) — the multi-worker process pool previously submitted one task per breakpoint, each independently reloading its own covariance partitions from disk with no sharing across concurrent workers. Peak memory scaled with in-flight *breakpoints*, not workers, and caused a real `BrokenProcessPool` OOM crash on a production run. Fixed by grouping breakpoints that share partition bounds (reusing the existing single-worker grouping logic) and submitting one task per group instead of one per breakpoint. See `docs/optimizations.md` #5.

2. **Filter-width search thread-parallelization** (`find_minima.py`) — profiling a real chr21 run (`plots/EUR-chr21-timeline.pdf`) surfaced an unlabeled, fully single-threaded ~34-of-64-second span that turned out to be `custom_binary_search_with_trackback`: ~41 sequential calls to a direct (non-FFT) Hanning convolution, each ~1s at realistic widths. Two of its three phases (exponential search's doubling, and the coarse/fine trackback refinement sweep) evaluate a boundable, predictable set of candidates per round and are now thread-parallelized — same decision logic, same numerics, computed concurrently instead of one-at-a-time. The core binary search remains sequential (adaptive, shared utility — see below). See `docs/optimizations.md` #11.

3. **numba direct convolution** (`filters.py`) — replaced `scipy.ndimage.convolve1d` with a hand-written direct convolution compiled via `@njit(nogil=True, fastmath=True, cache=True)`. Same O(N·width) algorithm and shift-invariant summation structure as scipy's (flat-region-safe by construction, unlike FFT below), but ~2x faster per call — multiplicative with #2's threading for a combined ~10x over the original sequential-scipy baseline. This is the item marked "still open" below in an earlier pass of this investigation; now done. `fastmath=True` turned out to be load-bearing, not optional — without it the compiled kernel was ~2x *slower* than scipy (LLVM didn't auto-vectorize the strict-order reduction loop), and confirming it didn't reopen the FFT-style risk required its own verification (see below). See `docs/optimizations.md` #12.

4. **BLAS/OMP thread-count guard** (`examples/ldetect_original/Snakefile`, `_cli/cmd_run.py`) — found while diagnosing apparent multi-worker regressions during real-cluster validation (see below). Numpy/BLAS/numba size their own internal thread pools to the *whole machine's* core count by default, not `--workers`, unless `OMP_NUM_THREADS`/`OPENBLAS_NUM_THREADS`/`MKL_NUM_THREADS`/`NUMEXPR_NUM_THREADS`/`NUMBA_NUM_THREADS` are set. Harmless when a job has a node to itself; a real oversubscription risk under Slurm-style shared-node scheduling, where many `ldetect run` processes land on the same physical cores simultaneously and each independently assumes it owns every core. Fixed by exporting all five to `{params.workers}` in the example Snakefile's `run_ldetect` rule (reliable — a fresh subprocess per job, so no import-order concerns). `ldetect run` itself can't safely force this the same way: `_cli/main.py::main()` imports numpy-dependent submodules *before* `argparse` parses `--workers`, so by the time the CLI knows the worker count, BLAS has already initialized its thread pool from the ambient environment. Added a startup warning instead (`_run` in `cmd_run.py`) when `--workers > 1` and none of the five env vars are set, so anyone invoking `ldetect run` directly (outside this Snakefile) gets nudged rather than silently oversubscribed.

## Tried and rejected: FFT convolution

Swapping the filter's `scipy.ndimage.convolve1d` (O(N·width) direct convolution — the actual root cause of the ~1s/call cost) for `scipy.signal.fftconvolve` (O(N log N)) looked like the highest-leverage fix and was ~100-150x faster in isolation. **It is numerically unsafe for this pipeline and was reverted.**

Direct convolution produces bit-identical output across any flat/constant stretch of the input vector (same shift-invariant arithmetic repeated at every position), so the downstream strict-inequality minima detector (`argrelextrema`) never fires there. FFT convolution's rounding error is *not* shift-invariant — it injects distinct floating-point noise (~1e-15) at every position, which breaks exact ties and manufactures spurious local minima out of nothing (confirmed: 1 real minimum became 23 detected on a synthetic flat-plateau test signal). Since the search is actively hunting for a width producing an exact target minima count, spurious minima anywhere could derail convergence to a materially different breakpoint set — and real covariance-sum vectors can plausibly have flat/tied runs (sparse regions, zero-padding), so this isn't just a synthetic-fixture edge case. Other FFT-family methods (`oaconvolve`) share the same flaw.

## A subtler, more general finding: near-tied minima counts aren't only an FFT problem

Validating the numba kernel surfaced a narrower but more fundamental instability than the FFT bug, worth understanding on its own terms. A pre-existing test (`test_apply_filter_larger_width_fewer_minima`) failed under the numba kernel — not from a bug (the smoothed array matched scipy to ~1e-15, and the flat-plateau fixtures still passed exactly), but because its two compared widths sat at a point where **scipy's own result was already an exact tie** (20 minima at both), not a real margin. Sweeping widths 2 through 95 under plain scipy showed minima count is genuinely non-monotonic at intermediate widths for that periodic test signal (20, 18, 15, 21, 25, 24, 20, 18, 9, 3 — only trending down once the widths are decisively far apart).

The general lesson: **any two numerically-non-identical implementations of the same nominal algorithm — not specific to numba, and not requiring FFT-style global noise — can disagree on an exact minima count** whenever the smoothed *output* has near-zero local variation arising from the signal's own structure (as opposed to the FFT bug's mechanism: noise on a genuinely constant *input*). This would in principle also occur between two scipy versions or BLAS builds computing the "same" convolution. It's a real, narrower category of risk than FFT's, and was checked empirically rather than assumed away: 0 mismatches across 80 randomized trials of noisy random-walk vectors (structurally closer to real covariance-sum vectors than the adversarial periodic test signal) at production-representative widths up to 9,000. The failing test was fixed by choosing a decisive (non-tied) width pair, not by weakening its assertion.

## Real-cluster full-replication validation

Ran the full 22-chromosome x 3-population replication on both `main` and this branch (`profiling_main`/`profiling_cyvcf2` directories, Slurm `--cores 64`) and compared against a single-chromosome deep dive (EUR chr21, phase-by-phase from the `ldetect run` logs).

- **chr21 in isolation: 4.1x faster end-to-end** (281s -> 69s), every phase improved (local search alone: 115s -> 10s).
- **Full replication (66 rows): net positive but noisier.** Median wall-clock 17% faster, total -9.7% (~137 min saved); core utilization up (median +0.28 cores). Peak RSS went *up* in aggregate (median +14%), not down — traced to `calc_covariance` moving VCF decompression in-process (`cyvcf2`) where `main` shelled out to a separate `tabix` subprocess per partition; some of the RSS increase is memory that always existed but is now attributed to the measured process instead of an external child.
- **Two rows (AFR chr4, chr11) initially looked like regressions** (slower wall-clock, *lower* core utilization than the general trend) — traced via `.ldetect.log` timestamps to Slurm scheduling both chromosomes' jobs starting at the identical second, i.e. genuine machine contention, not a code regression: unrelated, untouched phases (`step3`, `fourier_metric`) were also 7-8x slower under contention. Confirmed by re-running AFR chr4 in isolation under a 4-CPU cap: 25.3 min, a clean 1.7x win over `main`'s 43.5 min.
- That isolated re-run surfaced item #4 above (BLAS/OMP thread guard): a single ~1-2s sample hit `cpu_percent_sum=1905%` on a 4-CPU-capped job, timed exactly to `_run_local_search`'s `ProcessPoolExecutor` startup. Consistent with BLAS/numba worker processes each sizing thread pools to the full node rather than the 4-CPU allocation — a plausible amplifier of the chr4/chr11 contention above when many such jobs share one Slurm allocation.

## Still open

- **Binary-search phase** (`_util/binary_search.py::find_le_ind`) is not parallelized — each step is adaptive on the previous comparison, so it can't be pre-batched the same way as the other two `find_minima.py` phases without a fundamentally different (k-ary search) algorithm. It's also a shared utility used elsewhere, so changing its core algorithm needs more care/validation than was in scope this pass. It does now benefit from the numba kernel's ~2x per-call speedup even though it isn't itself parallelized.
- **`numba.prange`-parallel convolution** (within-call multicore, on top of the per-call speedup already shipped) — deferred. Gating it to only the phases with no outer `ThreadPoolExecutor` concurrency (to avoid oversubscription during `_trackback`) would require `custom_binary_search_with_trackback` to use a *different* filter function per phase, touching `find_minima.py`'s shared `f: Callable` plumbing, `FlexibleBoundedAccessor`, and `pipeline.py` — a multi-file refactor for a secondary win, since the per-call fix already shipped is the dominant cost reduction. `parallel=True`/`prange` is used nowhere else in the repo currently; would be a first if picked up.
- **Cross-stage pipelining** (covariance calc → matrix-to-vector → local search running concurrently across stage boundaries) — considered and deprioritized: step 4's binary search needs the complete chromosome vector before it can start, so the only real overlap opportunity is the "tail" of step 2 (idle workers waiting for the last few partitions), which is a narrower win than it first sounds.
- Real-cluster wall-clock validation is now done (see above) and confirms a real win, but the full-replication run that produced the aggregate numbers used `--local-search-workers 1` throughout (both `main` and branch) — meaning the local-search multi-worker/OOM-grouping fix itself was never actually exercised in that comparison; the measured aggregate speedup is attributable to the cyvcf2 I/O migration and the numba per-call convolution speedup alone. A clean re-run with `--local-search-workers` > 1 and the new BLAS thread caps in place (isolated per chromosome, or at least confirming Slurm isn't co-scheduling large chromosomes together) is still needed to validate the multi-worker local-search path at production scale.
- Whether the BLAS/OMP thread-cap fix (item #4) actually resolves the chr4/chr11-style contention needs a follow-up full-replication run to confirm — the isolated single-chromosome re-run validates the *mechanism* (the CPU spike lines up with pool creation) but not yet the fix's effect on a genuinely concurrent Slurm allocation.
